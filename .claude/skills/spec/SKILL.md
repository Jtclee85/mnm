# spec — 작업지시서 실행 프로토콜

## 트리거
- "N차 구조 변경 작업", "이번 작업의 범위는 …", "아래 작업을 하지 마라" 같은 구조화된 지시서.

## 입력
- 사용자의 작업지시서 전문.

## 단계

1. **범위 재진술**: 이번 작업에서 할 것 / 하지 않을 것을 각각 목록으로 재진술한다.
2. **금지 체크리스트 고정**: 지시서의 "하지 마라" 항목을 체크리스트로 만들어 구현 내내 유지한다.
3. **영향 파일 목록**: 수정할 파일과 이유를 나열한다. `pages/index.js`에 새 기능을 쌓지 말고 `components/` 분리를 우선 검토한다.
4. **구현**: 최소 침습으로 수정한다. 지시서에 없는 리팩토링·개선을 끼워 넣지 않는다.
5. **범위 자기검사**: 구현 후 `git diff --stat`을 보고 (a) 범위 밖 파일 변경 0건 (b) "하지 마라" 항목 위반 0건을 확인해 결과를 명시한다.
6. **테스트**: CLAUDE.md "완료의 정의" 게이트를 실행한다.
7. **요약**: 변경사항을 지시서의 항목 번호에 대응시켜 보고한다.

## 출력
- 범위 재진술 → 구현 → 자기검사 결과 → 테스트 결과 → 지시서 항목별 대응 요약.
- 지시서가 이전 차수 작업을 전제하면, 코드에서 그 전제가 실제로 충족돼 있는지 먼저 확인하고 어긋나면 작업 전에 보고한다.
- 이미 완료된 지시인지 애매하면 (사용자가 "내가 했던가?"라고 물을 때 포함) git log와 코드를 먼저 확인해 답한다.
