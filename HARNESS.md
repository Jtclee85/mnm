# HARNESS.md
Project: 뭐냐면
Purpose: 깊이있는 학습 지원을 위한 AI 기반 조사자료 재구성 웹앱

---

## 1. 프로젝트 정체성

- 「뭐냐면」은 단순 AI 챗봇이 아니라 깊이있는 학습을 지원하는 AI 기반 조사자료 재구성 웹앱이다.
- 초등 사회과 조사학습, 박물관 학습, 전시 설명문 읽기, 안내문 이해 등을 지원한다.
- AI는 학생 대신 학습하지 않는다.
- AI는 학생이 생각하도록 돕는다.
- 교육용 SW·AI 연구대회 출품과 실제 학교 현장 활용을 목표로 한다.
- 주 사용 환경은 1920×1080 해상도의 PC/교실 환경이다.
- 모바일은 보조적으로 지원한다.

---

## 2. 핵심 교육 철학

**이해 → 탐구 → 표현 → 성찰**

- 새 기능은 반드시 이 네 단계 중 하나 이상을 지원해야 한다.
- 학생의 사고를 줄이는 기능이 아니라 학생의 사고를 늘리는 기능을 우선한다.
- AI 결과물은 학생의 사고를 돕는 발판이어야 한다.
- AI 결과물이 학생의 최종 산출물을 대체해서는 안 된다.
- 학생이 직접 쓴 생각, 질문, 설명, 성찰이 AI 결과물보다 더 중요하다.

---

## 3. 깊이있는 학습 원칙

- **개념적 이해**: 단편적 정보가 아니라 개념을 이해하도록 돕는다.
- **탐구 질문**: 단순 정답 찾기가 아니라 사고를 확장해야 한다.
- **학습자 주도성**: AI가 대신 완성해주는 방식보다 학생이 직접 작성하도록 유도하는 방식을 우선한다.
- **삶과 연계**: 학생의 경험·생활과 연결되도록 돕는다.
- **전이**: 배운 내용을 다른 상황에 적용할 수 있도록 돕는다.
- **성찰**: 학생이 자기 말로 설명하고, 질문하고, 성찰하도록 돕는 구조를 우선한다.
- **학생 산출물**: 학생이 직접 만든 결과물(요약, 질문, 발표문, 성찰)을 향후 핵심 기능으로 발전시킨다.

## 5. 모드별 설계 원칙

### 이해 모드
- 어려운 자료를 학생 수준에 맞게 이해하도록 돕는다.
- 쉬운 설명, 낱말 풀이, 핵심 내용 정리에 초점을 둔다.
- 학생이 원문을 포기하지 않고 다시 접근할 수 있게 돕는다.
- 원문의 의미를 왜곡하지 않는다.

### 탐구 모드
- 정답 찾기보다 질문 만들기와 사고 확장에 초점을 둔다.
- 탐구 질문은 개방성, 토론 가능성, 전이 가능성, 높은 수준의 사고를 유도해야 한다.
- 추천 검색어는 학생의 추가 조사를 돕는 보조 장치로 사용한다.
- AI가 탐구를 대신 끝내는 것이 아니라 학생이 더 조사하도록 돕는다.

### 발표 준비 모드
- 학생이 조사한 내용을 다른 사람에게 설명할 수 있도록 돕는다.
- 발표 제목, 발표 구조, 예상 질문, 핵심 메시지를 지원한다.
- 발표문을 대신 완성해주는 기능이 아니라 학생의 표현을 돕는 구조여야 한다.
- 학생이 자기 말로 발표하도록 유도한다.

### 쉬운한국어 모드

쉬운한국어 모드는 **이미 구현된 핵심 모드**다. 한국어 사용 수준이 낮은 다문화 학습자, KSL 초기 학습자, 유치원~초1 수준의 한국어 이해가 필요한 학생을 위한 모드다.

단순히 "짧게 요약"하는 기능이 아니라, **원문의 의미를 유지하면서 언어 난도를 크게 낮추는 기능**이다.

반드시 지켜야 할 원칙:

- 한 문장에는 한 가지 정보만 담는다.
- 문장은 가능하면 30자 이내로 짧게 작성한다.
- 쉬운 낱말을 사용한다.
- 어려운 한자어와 추상어를 줄인다.
- 어려운 낱말은 바로 풀어서 설명한다.
- 수동 표현보다 능동 표현을 사용한다.
- 대명사 사용을 줄이고, 핵심어를 반복해도 좋다.
- "~해요", "~예요" 문체를 사용한다.
- 의미를 삭제하거나 왜곡하지 않고, 난도만 낮춘다.
- 어린아이 말투로 과도하게 단순화하지 않는다.
- 역사적·사회적 사실을 틀리게 바꾸지 않는다.
- 쉬운한국어 모드를 단순 요약 기능으로 축소하지 않는다.

쉬운한국어 모드는 앞으로 테스트 하네스에서 반드시 보호해야 할 핵심 기능이다.

## 6. UI/UX 원칙

- 주 사용 환경은 1920×1080 해상도의 PC/교실 환경이다.
- 모바일도 지원하지만, 1순위 기준은 PC 수업 환경이다.
- PC에서는 좌우 2단 또는 넓은 카드형 레이아웃을 안정적으로 유지한다.
- 모바일에서는 깨지지 않고 1단으로 자연스럽게 전환된다 (현재는 결과 캔버스가 풀스크린 오버레이로 전환되는 방식).
- 교사와 학생이 전자칠판 또는 교실 PC 화면에서 볼 수 있도록 가독성을 확보한다.
- 카드형 UI를 유지한다.
- `SectionCard`와 기존 컴포넌트 구조를 존중한다.
- 지나치게 복잡한 UI를 피한다.
- 초등학생과 교사가 직관적으로 사용할 수 있게 한다.
- 새로운 기능은 기존 학습 흐름을 방해하지 않아야 한다.
- 자료조사 도움말이나 외부 링크 패널을 추가할 경우, PC에서는 사이드 패널을 우선 고려하고 모바일에서는 접이식 구조를 고려한다.

## 7. 코드 구조 원칙

현재 구조:

```
pages/index.js          메인 화면 (입력 폼, 좌우 레이아웃, 상태 관리)
pages/share.js           교사용 결과 공유 페이지 (한국어 고정)
pages/api/chat.js        OpenAI 스트리밍 채팅 API
pages/api/relevance.js   후속 질문 관련성 판단 API
components/              화면을 구성하는 프레젠테이션 컴포넌트
lib/systemPrompt.js      AI 시스템 프롬프트 생성
lib/parseResponse.js     AI 응답 파싱 (태그 기반)
lib/i18n.js              다국어 UI 텍스트 + 언어별 프롬프트 지시문
lib/useSessionSave.js    이전 조사 저장/불러오기 (localStorage)
lib/useStudentNotes.js   학생 메모 저장 (localStorage)
lib/recommendedSources.js  추천 원본자료 데이터 (JSX 외부 분리)
tests/                   Playwright E2E 테스트
```

원칙:

- 새 UI는 가능한 `components/`에 컴포넌트로 분리한다.
- 프롬프트 관련 코드는 가능한 `lib/systemPrompt.js`에서 관리한다.
- 응답 파싱 관련 코드는 가능한 `lib/parseResponse.js`에서 관리한다.
- 큰 파일(`pages/index.js` 등)에 모든 기능을 계속 추가하지 않는다.
- 기존 기능을 깨지 않도록 최소 침습적으로 수정한다.
- 불필요한 라이브러리 추가를 피한다.
- 중복 코드를 만들지 않고 재사용 가능한 구조를 우선한다.
- 쉬운한국어 모드 관련 로직은 다른 모드와 충돌하지 않도록 독립성과 일관성을 유지한다.
- 외부 링크나 교육자료 데이터는 가능하면 JSX에 직접 박지 말고 별도 데이터 파일로 분리한다 (`lib/recommendedSources.js` 참고).

## 8. AI 작업 절차

Claude Code(또는 다른 AI 코딩 도구)는 작업 시 다음 순서를 따른다.

1. 작업 목표 재진술
2. 구현 계획 설명
3. 영향받는 파일 목록 설명
4. 코드 수정
5. 테스트 실행
6. 변경사항 요약
7. commit/push 제안

추가 원칙:

- 큰 구조 변경이 필요한 경우에는 바로 수정하지 말고 먼저 사용자에게 확인을 요청한다.
- 기존 기능을 제거하거나 UI를 크게 바꾸는 경우에는 반드시 사전 확인을 받는다.
- 테스트가 실패하면 실패 원인을 설명하고 수정한다.
- 테스트를 실행할 수 없다면 그 이유를 설명한다.

## 9. 테스트 하네스 원칙

- Playwright는 실제 사용자 흐름 검증용이다.
- Vitest는 내부 로직 검증용이다 (현재 미설치 — 향후 `lib/parseResponse.js`, `lib/i18n.js` 등 순수 로직 검증에 도입 예정).
- 새 기능을 추가하면 관련 테스트를 함께 추가하거나 기존 테스트 영향 여부를 확인한다.
- 실제 OpenAI API를 테스트에서 직접 호출하지 않고 mock 응답(`page.route('**/api/chat', ...)`)을 우선 사용한다.
- 1920×1080 데스크톱 테스트를 1순위로 둔다 ([playwright.config.js](playwright.config.js) 기본 viewport, [tests/smoke.spec.js](tests/smoke.spec.js)).
- 모바일 테스트는 보조 회귀 방지용으로 둔다 ([tests/mobile.spec.js](tests/mobile.spec.js)).
- 테스트 통과 전에는 commit/push를 제안하지 않는다.

명령어:

- `npm run dev` — 개발 서버 실행 (현재 존재)
- `npm run test:e2e` — Playwright E2E 테스트 실행 (현재 존재)
- `npm run test:unit` — Vitest 단위 테스트 실행 (현재 미존재, Vitest 도입 시 추가)
- `npm run test:e2e:ui` — Playwright UI 모드 실행 (현재 미존재, 필요 시 `playwright test --ui` 별칭으로 추가)

### 쉬운한국어 모드 테스트 원칙

- 쉬운한국어 모드를 선택할 수 있는가
- 쉬운한국어 모드 분석 결과 카드가 정상적으로 표시되는가
- mock 응답을 사용했을 때 짧은 문장 중심 결과가 표시되는가
- 어려운 낱말이 쉬운 말로 풀리는가
- 기존 이해/탐구/발표 준비 모드가 깨지지 않는가
- PC 1920×1080 화면에서 결과 카드가 안정적으로 보이는가
- 모바일 화면에서도 레이아웃이 깨지지 않는가

### 자료조사 도움말/외부 링크 패널 테스트 원칙

향후 자료조사 도움말, 외부 링크, 교육자료 사이드 패널을 추가/확장할 경우 다음을 확인한다 (현재 `RecommendedSources` 배너가 그 1차 구현이다):

- 도움말(또는 패널) 열기 버튼이 보이는가
- 패널이 열리고 닫히는가
- PC 1920×1080 화면에서 패널이 본문을 가리지 않는가
- 모바일 화면에서 접이식 구조가 깨지지 않는가
- 모든 외부 링크에 `href`가 있는가
- 외부 링크는 새 탭으로 열리는가
- 외부 링크에는 `target="_blank"`와 `rel="noopener noreferrer"`가 있는가
- 모든 자료 카드에 제목, 설명, 출처 또는 링크가 있는가
- 빈 링크(`#`)가 없는가

## 10. Git 작업 원칙

- 큰 기능은 새 브랜치에서 작업한다.
- 작업 전 `git status`를 확인한다.
- 작업 완료 후 변경 파일을 요약한다.
- commit 메시지는 의미 있게 작성한다.
- 테스트 통과 후 commit/push를 제안한다.
- commit과 push는 사용자의 확인 후 진행한다.

좋은 commit 예시:
- `feat: 쉬운한국어 모드 추가`
- `feat: 학생 산출물 입력 기능 추가`
- `fix: 퀴즈 선택지 클릭 오류 수정`
- `test: Playwright 스모크 테스트 추가`
- `test: 쉬운한국어 모드 E2E 테스트 추가`
- `docs: HARNESS 문서 추가`

나쁜 commit 예시:
- `update`
- `fix`
- `수정`
- `작업`

## 11. 절대 금지 사항

- 학생 대신 탐구보고서를 완성해주는 기능
- 학생 대신 수행평가 산출물을 작성해주는 기능
- 정답만 제공하고 사고를 요구하지 않는 기능
- 기존 기능을 이유 없이 삭제하는 수정
- UI 전체를 사용자 동의 없이 갈아엎는 수정
- 테스트 없이 큰 변경을 push하는 작업
- API 키나 민감정보를 코드에 직접 넣는 작업
- 쉬운한국어 모드에서 원문의 사실을 왜곡하는 수정
- 쉬운한국어 모드를 단순 요약 기능으로 축소하는 수정
- 실제 OpenAI API를 테스트에서 반복 호출하는 구조

## 15. 최종 목표

「뭐냐면」은 "조사자료를 쉽게 바꾸는 앱"에서 멈추지 않고, "깊이있는 학습을 지원하는 AI 기반 사회과 조사학습 플랫폼"으로 성장하는 것을 목표로 한다.

현재의 쉬운한국어 모드는 이 목표 안에서 다문화 학습자와 한국어 학습 지원이 필요한 학생의 자료 접근성을 높이는 핵심 기능으로 유지·발전시킨다.
